# 做一个会陪我上班的桌宠

## 起因

在codex把桌宠做出来之前，我就一直想做一个桌面陪伴的应用，因为工作本身已经比较严肃烧脑了，可以适当来点整活的、有意思的小东西添添乐子。我想要一个可爱的、可以复刻我家人样子的 Q 版小角色，不太大、不抢镜，安安静静地待在屏幕角落，偶尔走两步、偶尔睡觉、偶尔开心地蹦跳一下。让上班这件事多一点温度，让我工作时的思念有一份落脚处。

情绪价值这种事很难量化，但很重要。

---

## 规划

一开始我就想，要做就做轻一点。

我不打算一上来就堆功能。对话能力、实时反应、AI 助手 这些都是后面的事。首期我准备先把最核心的一件事做好就够了：让用户上传一张照片，就能得到这个人的 Q 版形象，并且能在桌面上自然地动起来。
所以最先要解决的问题是图像生成。普通的文生图模型给一张参考照片，画出来的人"一眼就不是那个人"，那种东西放在桌面上是没有意义的——我做这件事的意义就在于"是这个人"。后来反复对比，最理想的是 GPT-Image-2，它对人物特征的保真度很高，同一个人在不同动作下还能认得出来还是同一个人。当时这个模型刚出来不久，为了图便宜我最后通过 Evolink 中转通道接上了，接口规格基本一致。

模型选好之后，我开始用 Claude Code 的 internal 版本叠加 CodeBuddy IDE，一边聊一边改，正式动手。

---

## 让静态图先动起来

模型生成的是单张图片，不是动画。要让它"动"，得有帧。

我的做法是让模型一次性画出整张精灵图（sprite sheet）：在 prompt 里讲清楚要画 8 个连续姿势、横向排成一排、3840×1280 分辨率、每帧 480 像素宽（绿幕背景，方便抠图）。生成完之后再用 Canvas 按等宽切成 8 张独立的 PNG，分别落到对应动作的文件夹下。

第一次跑通的时候我上传的是一张自己的照片。等了几分钟，屏幕上出现了一个戴马尾、挎着黄色小包的 Q 版小女孩，站在透明背景上做着 idle 动作——眨眼、转头、左右晃，动作还挺像那么回事，就是看着比本人可爱多了。

---

## 让它有自己的情绪

光有帧只能让单个动作循环播放，要让它真的"活"起来，得有一套状态机决定它什么时候在做什么。

我希望它有自己的小情绪。鼠标长时间不动的时候，它会自己走起来散步；走累了会偶尔停下张望四周；很久没人理它就进入睡眠；睡得太久偶尔会变成"难过"的表情；点它一下会蹦跳着表示开心；想拖动它的时候会摆出委屈脸；只要用户重新开始活动，它就会自动回到默认状态。看起来很像一只小猫的作息。

虽然写一个状态机本身并不难，但边界情况判定还是有点烦人。

让我印象最深的一个 bug 是这样的：走路的时候右键打开菜单切换动作，桌宠会瞬间往右下角跳 220 像素。我调了一下午才搞清楚根因——打开右键菜单的时候，窗口本身需要扩大并向左上偏移 220px 的 padding，这样视觉上桌宠不动、菜单从桌宠右侧弹出来才有空间。但走路用的 `requestAnimationFrame` 循环还在跑，每一帧都把窗口位置覆写回原来的坐标，结果桌宠就落在了一个"带着 padding 的窗口里、又偏移了整整 220px"的错位位置上。

修复方案最后只有几行——打开菜单前先把走路停掉，关菜单后如果状态仍是 walk 再恢复。但找到这几行的过程，是我先后看着桌宠在屏幕上瞬移了几十次。这是异步、setPosition、React state 三股力量在同一个时间窗口里抢占窗口位置造成的事故。类似的隐性边界后面又遇到过好几个，每一个都要花时间慢慢摸清楚。

开发那段时间，这只桌宠在我屏幕上演过的奇怪姿势——瞬移、抽搐、横着飘、卡在 Dock 里、动作切到一半凝固——比它最后正式上线会做的动作集还要丰富一些。。。。

---

## 不只是我一个人

既然能生成"我自己"的 Q 版形象，那当然也可以是家人、朋友、宠物。我做了一个形象管理面板，让所有形象都能并存：上传不同的照片可以生成多个独立的形象，每个形象都有自己的动作帧；右键菜单可以一键切换；每个形象都可以重命名、可以删除；文件结构是一形象一文件夹，互相不打扰，存放在 `~/Library/Application Support/com.desktop-pet.app/avatars/` 下面。

我后来一口气测试了好几个形象——一个戴马尾的小女孩、一个戴帽子喝咖啡的老爷爷、一只表情慵懒的橘猫、还有初音未来。每一个跑出来都比想象中要可爱。这种"任何照片都能变成 Q 版"的感觉，比我最初设想的应用场景要丰富很多。

---

## 给它做一个家

App 完成之后总得有个让人能下载、能了解的地方。我做了一个简单的着陆页，用 Vercel 部署在 [my-pet-heart.vercel.app](https://my-pet-heart.vercel.app/)。

整个网站延续了 App 本身的暖墨手账风格，整屏 snap-scroll 吸附往下滚，每一屏只讲一件事。首屏一句话标语配着一只大角色，旁边摆着 7 个动作小卡——鼠标 hover 上去哪一个，主角就实时切换播放对应的动画，像在跟你打招呼。第二屏是手绘 SVG 画的三步流程：上传照片、AI 绘制、桌面陪伴。第三屏是 5 张稍稍倾斜散落的拍立得卡片，每一张里都循环播放着一个角色的 idle 帧——小女孩、老爷爷、小猫、初音未来、emoji 占位——专门用来打消"是不是只能用作者自己的形象"的疑虑。第四屏模拟了一个 Mac 桌面，桌宠在右下角自动演示几种行为，闲置走两步、不动了去睡觉、点击会开心、走累了张望一下。第五屏是下载入口。

为了让拍立得卡里的每一个角色都真的"活着"，我让每张卡片在 mount 的时候一次性把所有帧预加载到浏览器缓存里，避免切帧的瞬间闪烁。

整个网站还藏了一些不显眼的小心思——主标题下面是手绘风的波浪线下划线，背景里飘着小星星和小心，卡片轻微旋转着 1-2°（像贴纸贴歪了），鼠标 hover 时它们会回正、轻轻抬起一点点。每一个细节都不大，但叠在一起，整个网站就有了一种"案头小物件"的感觉，而不是又一个标准的 SaaS 落地页。

## 关于技术选型

简单聊几句。桌宠这类应用的需求其实有点奇怪——它要常驻桌面但不能像普通 App 那样在 Dock 里占一个图标，要透明背景让形象抠图后能贴在桌面上，要让鼠标点击穿透过去（点桌面其他地方不能被挡），要永远置顶但又不能霸屏。最关键的是它得轻——毕竟这东西要 24 小时挂着的。

如果用 Electron，光启动就要吃掉几十 MB 内存，跟我"陪伴"的初衷完全背道而驰，反而成了一种负担。所以最后选了 Tauri，Rust 加 WebView 的组合。前端继续用 React 加 TypeScript（写得最熟），状态机、动画、UI 这些全在 webview 里跑；窗口管理、跨工作区显示、把窗口 level 设到 1500 让它浮在所有空间之上、监听用户空闲时间这些跟系统打交道的事，交给 Rust 处理。

最终打包出来的 .dmg 只有 7.2 MB。这个数字让我满意了一会儿。

着陆页是独立项目独立部署，技术栈一致：Vite、React、TypeScript，没有什么特别的理由——就是写起来顺手。

## 写到这里

整个项目我花了周五一个晚上加周六白天来做。
我后来想了想为什么自己愿意在一只屏幕上的小角色身上花这么多时间。

大概就是，我少女时代的英雄主义，总是想成为一个可以靠写代码实现自己想法的人，总是去为人们的需求创造出解决方案。那时候这件事看起来还很远。

现在，我终于成为了一个可以靠自己写的代码实现自己想法的人了。
